## 1. Search Engine – Elasticsearch vs. Prisma/SQLite for MVP

**✅ I fully approve using Prisma + PostgreSQL (or SQLite for early dev) for the MVP.**

- **Why it’s safe:** Your MVP will have at most a few thousand vendors. PostgreSQL’s built‑in full‑text search (`tsvector`/`tsquery`) and Prisma’s filtering (`where`, `orderBy`, `take`/`skip`) are more than enough for:
  - Category + location + budget range + rating + response time filters
  - Sorting by rating, price, response time
  - Pagination (20 per page)
- **Avoid premature complexity:** Elasticsearch adds operational overhead (cluster setup, indexing, sync with Prisma). You can add it in Phase 2 (after MVP validation) when you have >10k vendors or need fuzzy search / “Best Match” scoring.
- **SQLite caution:** For local development SQLite is fine, but for production MVP use **PostgreSQL** (even on a small VM or free tier of Render/Supabase). SQLite’s concurrent write limitations and weaker full‑text search will become painful quickly.

**Action:** Use Prisma with PostgreSQL. Implement filters as Prisma `where` clauses. For “Best Match” sorting, a simple formula (e.g., `rating * 0.4 + responseScore * 0.3 + locationMatch * 0.3`) can be computed in JavaScript after fetching – or store a pre‑computed `matchScore` column if needed.

**Mock Elasticsearch?** No need. Build the real filtering logic with Prisma. It will work and be easy to replace later with Elasticsearch (just swap the repository layer).
